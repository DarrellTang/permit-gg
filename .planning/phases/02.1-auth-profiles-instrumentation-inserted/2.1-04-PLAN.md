  <action>
The PostHog setup has three conflicting pageview mechanisms:
1. `capture_pageview: true` in the init config (auto-captures on page load)
2. A `loaded` callback that manually captures `$pageview`
3. A `PostHogPageView` component that captures `$pageview` on route changes

These conflicts can cause PostHog to enter a confused state or triple-fire pageviews, potentially suppressing events via rate limiting.

**Fix provider.tsx:**

Set `capture_pageview: false` (per the original research recommendation and the App Router pattern). Remove the `loaded` callback that manually fires a pageview. The `PostHogPageView` component handles pageviews on route changes, which is the correct approach for Next.js App Router (since automatic pageview tracking misses client-side navigations).

```typescript
"use client"

import posthog from "posthog-js"
import { PostHogProvider as PHProvider } from "posthog-js/react"

if (typeof window !== "undefined") {
  posthog.init("phc_G6G0W5qoY5uXOyAbA0674BW4ugdnYFZl6DstK7zUSPJ", {
    api_host: "/ingest",
    ui_host: "https://us.i.posthog.com",
    person_profiles: "always",
    capture_pageview: false,
    capture_pageleave: true,
    capture_exceptions: true,
    autocapture: true,
    persistence: "localStorage+cookie",
    debug: process.env.NODE_ENV === "development",
  })
}

export function PostHogProvider({ children }: { children: React.ReactNode }) {
  return <PHProvider client={posthog}>{children}</PHProvider>
}
```

Key changes:
- `capture_pageview: false` -- let PostHogPageView handle it (App Router compatible)
- Removed `loaded` callback with manual `$pageview` capture
- Added `debug: process.env.NODE_ENV === "development"` for local troubleshooting (no effect in production)
- Everything else stays the same (hardcoded key is fine since it's a public frontend key)
