**Bug B: saveQuizResults may fail for anon users (GAP 1 contributing cause)**

The `saveQuizResults` server action calls `supabase.auth.getUser()`. For an unauthenticated user, this returns `{ data: { user: null }, error: ... }`. Then it inserts with `user_id: null`. The anon INSERT policy (`WITH CHECK (user_id IS NULL)`) should allow this. BUT: the server-side Supabase client from `createClient()` in server.ts uses the request cookies. If no auth cookie exists, the client operates as the `anon` role. The anon INSERT policy on quiz_sessions checks `user_id IS NULL` which should pass.

However, there could be a subtle issue: `supabase.auth.getUser()` for anon users might throw or return an error that gets mishandled. Add defensive error handling:

In `saveQuizResults`, wrap the getUser call defensively:
```typescript
let userId: string | null = null
try {
  const { data: { user } } = await supabase.auth.getUser()
  userId = user?.id ?? null
} catch {
  userId = null
}
```

```sql
-- Ensure anon policies exist for free quiz support
-- These may not have been applied in production

DROP POLICY IF EXISTS "Anon users can insert sessions" ON quiz_sessions;
DROP POLICY IF EXISTS "Anon users can insert answers" ON quiz_answers;
DROP POLICY IF EXISTS "Anon users can view anon sessions" ON quiz_sessions;
DROP POLICY IF EXISTS "Anon users can view anon answers" ON quiz_answers;

CREATE POLICY "Anon users can insert sessions"
  ON quiz_sessions FOR INSERT
  TO anon
  WITH CHECK (user_id IS NULL);

CREATE POLICY "Anon users can insert answers"
  ON quiz_answers FOR INSERT
  TO anon
  WITH CHECK (user_id IS NULL);

CREATE POLICY "Anon users can view anon sessions"
  ON quiz_sessions FOR SELECT
  TO anon
  USING (user_id IS NULL);

CREATE POLICY "Anon users can view anon answers"
  ON quiz_answers FOR SELECT
  TO anon
  USING (user_id IS NULL);
```
