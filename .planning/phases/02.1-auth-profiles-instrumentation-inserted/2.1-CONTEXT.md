# Phase 2.1: Auth, Profiles & Instrumentation - Context

**Gathered:** 2026-03-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Multi-user infrastructure: Supabase Auth with Google + Discord OAuth (email/password fallback), user profiles table with future-proof Stripe/tier columns, PostHog product analytics, and user-scoped quiz data. Does NOT include Stripe integration, paywalls, feature gating logic, or analytics dashboard — just the schema and auth wiring that enables those features later.

</domain>

<decisions>
## Implementation Decisions

### Auth Providers
- Google OAuth — primary, near-universal
- Discord OAuth — secondary, fits .gg gaming brand, popular with teens
- Email/password — fallback for users who don't want OAuth
- No Apple OAuth for now (requires Apple Developer account)

### Login/Signup Page
- Dedicated full-page at `/login` — not a modal
- OAuth buttons prominent at top (Google + Discord), email/password form below as fallback
- Cyberpunk-styled to match the app aesthetic
- Landing page "Start Studying" CTA links here
- After auth, redirect to `/dashboard`

### Unauthenticated Experience
- Landing page (`/`) is public — no auth required
- One free practice quiz allowed without signing up — user experiences value first (endowment effect)
- After the free quiz, prompt to sign up to save results and continue
- All other routes (`/dashboard`, `/simulated-test`, `/practice` after first use) require auth
- Redirect to `/login` with a return URL so they land back where they were trying to go

### Profiles Table
- `profiles` table extending Supabase `auth.users` (standard Supabase pattern)
- Columns: `id` (FK to auth.users), `display_name`, `avatar_url`, `stripe_customer_id` (nullable), `tier` (enum: 'free' | 'pro', default 'free'), `created_at`, `updated_at`
- Auto-create profile row on signup via Supabase database trigger
- `stripe_customer_id` and `tier` are schema-only — no Stripe integration code in this phase

### User-Scoped Data
- Add `user_id` column (FK to auth.users) to `quiz_sessions` table
- Update RLS policies: users can only read/write their own quiz sessions and answers
- Update `saveQuizResults` server action to include authenticated user's ID
- Update `fetchSessionSummary` to scope by user
- Backfill Lilly's existing quiz data with her user_id after she creates an account

### PostHog Integration
- `posthog-js` SDK installed, `<PostHogProvider>` in root layout
- Automatic page view tracking
- `posthog.identify()` called on login with user ID + email
- Full instrumentation — custom events:
  - `quiz_started` (mode, question_count)
  - `quiz_completed` (mode, score, percentage, time_ms)
  - `quiz_quit` (mode, questions_answered, score)
  - `sim_test_started`
  - `sim_test_passed` (score, percentage)
  - `answer_submitted` (is_correct, category_slug, streak)
  - `category_drill_started` (category_slug)
  - `streak_achieved` (streak_count)
  - `wrong_answer_reviewed`
  - `signup_completed` (provider: google/discord/email)
- PostHog project needs to be created in PostHog dashboard before deployment

### Claude's Discretion
- Supabase Auth helper setup (server/client)
- Auth middleware implementation pattern
- PostHog project key injection method
- Free quiz tracking mechanism (cookie/localStorage for "one free quiz" gate)
- Profile auto-creation trigger SQL
- Migration ordering and backfill strategy

</decisions>

<specifics>
## Specific Ideas

- OAuth buttons should feel like game login — big, colorful, one-click. Not a corporate form.
- Discord button should use Discord's brand purple — leans into the .gg energy
- The "one free quiz" is a marketing psychology play (endowment effect + foot-in-the-door): let them experience the cyberpunk quiz, celebrate a streak, see their radar chart, THEN ask them to sign up
- PostHog events should use snake_case naming convention for consistency
- The free quiz prompt after completion should feel like an invitation, not a wall — "Save your progress and track your improvement"

</specifics>
