## Summary

This phase adds multi-user infrastructure to a working single-user Next.js 16 quiz app. The three pillars are: (1) Supabase Auth with Google/Discord OAuth and email/password fallback using `@supabase/ssr` for cookie-based session management, (2) a `profiles` table auto-populated via PostgreSQL trigger on signup with future-proof Stripe/tier columns, and (3) PostHog product analytics with full custom event instrumentation.

The existing codebase already has `@supabase/ssr` 0.9 and `@supabase/supabase-js` 2.99 installed, plus working server/client Supabase factories. The browser client (`src/lib/supabase/client.ts`) currently uses `createClient` from `@supabase/supabase-js` directly instead of `createBrowserClient` from `@supabase/ssr` -- this must be updated for auth cookie handling. No middleware exists yet. Google and Discord OAuth are pre-configured in the Supabase dashboard. The `quiz_sessions` and `quiz_answers` tables have open RLS policies (`USING (true)`) with no `user_id` column.

**Primary recommendation:** Follow the standard Supabase SSR + Next.js middleware pattern exactly. Use `getClaims()` (not `getUser()`) in middleware for performance. Use PostHog's provider pattern with `capture_pageview: false` and a custom `PostHogPageView` component for App Router compatibility. Gate the free quiz with a `localStorage` flag checked client-side and a cookie checked server-side in middleware.

## Architecture Patterns

### Pattern 1: Supabase Auth Middleware (Token Refresh)

**What:** Next.js middleware that refreshes Supabase auth tokens on every request using `getClaims()`. Required because Server Components cannot write cookies.

**When to use:** Every request to the app (except static assets).

**Implementation:**

```typescript
// src/middleware.ts
import { type NextRequest, NextResponse } from 'next/server'
import { createServerClient } from '@supabase/ssr'

export async function middleware(request: NextRequest) {
  let response = NextResponse.next({ request })

  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return request.cookies.getAll()
        },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value, options }) => {
            request.cookies.set(name, value)
            response.cookies.set(name, value, options)
          })
        },
      },
    }
  )

  // Refresh auth token -- use getClaims() for performance (local JWT validation)
  // getUser() makes a network call every time; getClaims() only calls server when expiring
  const { data: { claims }, error } = await supabase.auth.getClaims()
  const user = claims // contains sub (user id), email, etc.

  // Route protection logic
  const isAuthRoute = request.nextUrl.pathname.startsWith('/dashboard') ||
    request.nextUrl.pathname.startsWith('/practice') ||
    request.nextUrl.pathname.startsWith('/simulated-test')

  if (isAuthRoute && (error || !user)) {
    const loginUrl = new URL('/login', request.url)
    loginUrl.searchParams.set('next', request.nextUrl.pathname)
    return NextResponse.redirect(loginUrl)
  }

  return response
}

export const config = {
  matcher: [
    '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)',
  ],
}
```

**Source:** [Supabase SSR Next.js docs](https://supabase.com/docs/guides/auth/server-side/nextjs), [getClaims API reference](https://supabase.com/docs/reference/javascript/auth-getclaims)

**Key decision: `getClaims()` vs `getUser()` in middleware:**
- `getClaims()` validates JWT locally using JWKS endpoint (often cached). Fast, no DB round-trip.
- `getUser()` calls auth server every time. Slower but detects banned/deleted users immediately.
- For this app (teen quiz app, no admin banning), `getClaims()` is the right choice.
- Use `getUser()` in server actions where you need fresh user data.

### Pattern 3: OAuth Callback Route (PKCE Flow)

**What:** OAuth providers redirect to a callback URL with an auth code. A route handler exchanges the code for a session.

**Implementation:**

```typescript
// src/app/auth/callback/route.ts
import { NextResponse } from 'next/server'
import { createClient } from '@/lib/supabase/server'

export async function GET(request: Request) {
  const { searchParams, origin } = new URL(request.url)
  const code = searchParams.get('code')
  let next = searchParams.get('next') ?? '/dashboard'

  if (!next.startsWith('/')) {
    next = '/dashboard'
  }

  if (code) {
    const supabase = await createClient()
    const { error } = await supabase.auth.exchangeCodeForSession(code)
    if (!error) {
      const forwardedHost = request.headers.get('x-forwarded-host')
      const isLocalEnv = process.env.NODE_ENV === 'development'
      if (isLocalEnv) {
        return NextResponse.redirect(`${origin}${next}`)
      } else if (forwardedHost) {
        return NextResponse.redirect(`https://${forwardedHost}${next}`)
      } else {
        return NextResponse.redirect(`${origin}${next}`)
      }
    }
  }

  return NextResponse.redirect(`${origin}/login?error=auth-code-error`)
}
```

**Source:** [Supabase Google OAuth docs](https://supabase.com/docs/guides/auth/social-login/auth-google)

### Pattern 5: Profiles Table with Auto-Create Trigger

**What:** A `profiles` table that automatically gets a row inserted when a user signs up via a PostgreSQL trigger.

```sql
-- Migration: create profiles table
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  display_name TEXT,
  avatar_url TEXT,
  stripe_customer_id TEXT,
  tier TEXT NOT NULL DEFAULT 'free' CHECK (tier IN ('free', 'pro')),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can view their own profile"
  ON profiles FOR SELECT
  TO authenticated
  USING (auth.uid() = id);

CREATE POLICY "Users can update their own profile"
  ON profiles FOR UPDATE
  TO authenticated
  USING (auth.uid() = id)
  WITH CHECK (auth.uid() = id);

-- Trigger function: auto-create profile on signup
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER SET search_path = ''
AS $$
BEGIN
  INSERT INTO public.profiles (id, display_name, avatar_url)
  VALUES (
    NEW.id,
    COALESCE(NEW.raw_user_meta_data ->> 'full_name', NEW.raw_user_meta_data ->> 'name', NEW.email),
    NEW.raw_user_meta_data ->> 'avatar_url'
  );
  RETURN NEW;
END;
$$;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW
  EXECUTE FUNCTION public.handle_new_user();

-- Updated_at auto-update trigger
CREATE OR REPLACE FUNCTION public.update_updated_at()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$;

CREATE TRIGGER profiles_updated_at
  BEFORE UPDATE ON profiles
  FOR EACH ROW
  EXECUTE FUNCTION public.update_updated_at();
```

**Key details:**
- `SECURITY DEFINER` with `SET search_path = ''` is required because the trigger runs on `auth.users` (auth schema) but inserts into `public.profiles`. The empty search path prevents privilege escalation.
- Google OAuth provides `full_name` and `avatar_url` in `raw_user_meta_data`.
- Discord OAuth provides `full_name` (or `username`) and `avatar_url` in `raw_user_meta_data`.
- Email signup can pass `display_name` via the `data` option in `signUp()`.

**Source:** [Supabase managing user data docs](https://supabase.com/docs/guides/auth/managing-user-data)

### Pattern 6: User-Scoped Quiz Data (Migration + RLS)

**What:** Add `user_id` to existing tables, update RLS policies to scope data per user.

```sql
-- Migration: add user_id to quiz tables
ALTER TABLE quiz_sessions ADD COLUMN user_id UUID REFERENCES auth.users(id);
ALTER TABLE quiz_answers ADD COLUMN user_id UUID REFERENCES auth.users(id);

-- Index for performance with RLS
CREATE INDEX idx_quiz_sessions_user_id ON quiz_sessions(user_id);
CREATE INDEX idx_quiz_answers_user_id ON quiz_answers(user_id);

-- Drop old open policies
DROP POLICY "Allow all access to quiz_sessions" ON quiz_sessions;
DROP POLICY "Allow all access to quiz_answers" ON quiz_answers;

-- New user-scoped policies for quiz_sessions
CREATE POLICY "Users can view their own sessions"
  ON quiz_sessions FOR SELECT
  TO authenticated
  USING (auth.uid() = user_id);

CREATE POLICY "Users can insert their own sessions"
  ON quiz_sessions FOR INSERT
  TO authenticated
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users can update their own sessions"
  ON quiz_sessions FOR UPDATE
  TO authenticated
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

-- New user-scoped policies for quiz_answers
CREATE POLICY "Users can view their own answers"
  ON quiz_answers FOR SELECT
  TO authenticated
  USING (auth.uid() = user_id);

CREATE POLICY "Users can insert their own answers"
  ON quiz_answers FOR INSERT
  TO authenticated
  WITH CHECK (auth.uid() = user_id);

-- Allow anon users to insert quiz sessions/answers for free quiz
CREATE POLICY "Anon users can insert sessions"
  ON quiz_sessions FOR INSERT
  TO anon
  WITH CHECK (user_id IS NULL);

CREATE POLICY "Anon users can insert answers"
  ON quiz_answers FOR INSERT
  TO anon
  WITH CHECK (user_id IS NULL);
```

**Important:** `user_id` is nullable initially to support the one free quiz for unauthenticated users and to avoid breaking existing data. Anon RLS policies allow inserts with `user_id IS NULL`.

**Source:** [Supabase RLS docs](https://supabase.com/docs/guides/database/postgres/row-level-security)

### Pattern 7: PostHog Provider + Page View Tracking

**What:** PostHog initialization with a provider pattern for App Router, custom page view tracking component, and user identification.

```typescript
// src/lib/posthog/provider.tsx
'use client'

import posthog from 'posthog-js'
import { PostHogProvider as PHProvider } from 'posthog-js/react'
import { useEffect } from 'react'

export function PostHogProvider({ children }: { children: React.ReactNode }) {
  useEffect(() => {
    posthog.init(process.env.NEXT_PUBLIC_POSTHOG_KEY!, {
      api_host: process.env.NEXT_PUBLIC_POSTHOG_HOST,
      capture_pageview: false, // We capture manually for App Router
      capture_pageleave: true,
    })
  }, [])

  return <PHProvider client={posthog}>{children}</PHProvider>
}
```

```typescript
// src/lib/posthog/page-view.tsx
'use client'

import { usePathname, useSearchParams } from 'next/navigation'
import { useEffect } from 'react'
import { usePostHog } from 'posthog-js/react'

export function PostHogPageView() {
  const pathname = usePathname()
  const searchParams = useSearchParams()
  const posthog = usePostHog()

  useEffect(() => {
    if (pathname && posthog) {
      let url = window.origin + pathname
      if (searchParams.toString()) {
        url = url + '?' + searchParams.toString()
      }
      posthog.capture('$pageview', { $current_url: url })
    }
  }, [pathname, searchParams, posthog])

  return null
}
```

```typescript
// User identification -- call after login/signup
import { usePostHog } from 'posthog-js/react'

// In a component or effect after auth state changes:
const posthog = usePostHog()
if (user) {
  posthog.identify(user.id, { email: user.email })
} else {
  posthog.reset() // Clear identity on logout
}
```

**Key decisions:**
- `capture_pageview: false` is required for App Router because automatic page view tracking fires before hydration and misses client-side navigations. The `PostHogPageView` component handles it correctly.
- Wrap `PostHogPageView` in a `Suspense` boundary because `useSearchParams()` requires it in Next.js App Router.

**Source:** [PostHog Next.js docs](https://posthog.com/docs/libraries/next-js), [PostHog App Router integration](https://reetesh.in/blog/posthog-integration-in-next.js-app-router)

### Pattern 8: Free Quiz Gating Mechanism

**What:** Allow one free practice quiz without signup, then require auth. Use localStorage for client-side UX and a cookie for server-side middleware enforcement.

**Approach:**
1. When an unauthenticated user completes their free quiz, set `localStorage.setItem('permit_free_quiz_used', 'true')` and set a cookie via a Server Action: `cookies().set('permit_free_quiz_used', 'true', { maxAge: 365 * 24 * 60 * 60 })`.
2. In middleware: if user is not authenticated AND the `permit_free_quiz_used` cookie exists AND they're trying to access `/practice`, redirect to `/login`.
3. The first visit to `/practice` (no cookie, no auth) is allowed through.
4. After login, the cookie becomes irrelevant since auth takes over.

**Why not just localStorage?** Middleware runs server-side and cannot read localStorage. Cookie provides server-side enforcement. localStorage provides instant client-side UX (show signup prompt without round-trip).

### Anti-Patterns to Avoid

- **Using `getSession()` in server code:** Deprecated for security. The session token could be tampered with since it comes from cookies. Use `getClaims()` (for speed) or `getUser()` (for freshness) instead.
- **Using `@supabase/auth-helpers-nextjs`:** Deprecated. Use `@supabase/ssr` only.
- **Checking auth in individual pages instead of middleware:** Middleware provides centralized route protection. Individual page checks are for getting user data, not for access control.
- **Initializing PostHog in a Server Component:** `posthog-js` is client-only. Always use `'use client'` directive.
- **Storing PostHog key in a server-only env var:** Must be `NEXT_PUBLIC_` prefixed for client-side access.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Auth session management | Custom JWT handling | `@supabase/ssr` middleware | Token refresh, cookie management, PKCE flow all handled |
| OAuth provider integration | Custom OAuth flows | `supabase.auth.signInWithOAuth()` | Provider config is in Supabase dashboard. One method call per provider |
| Profile auto-creation | Application-level signup hooks | PostgreSQL trigger on `auth.users` | Guarantees profile exists regardless of signup path (OAuth, email, etc.) |
| Row-level data isolation | Application-level query filters | Supabase RLS with `auth.uid()` | Database-level enforcement. Cannot be bypassed by client code |
| Page view tracking in App Router | Custom router event listeners | PostHog `PostHogPageView` component | Handles client-side navigation, avoids double-fires |

## Common Pitfalls

### Pitfall 1: Forgetting to Add Redirect URLs in Supabase Dashboard
**What goes wrong:** OAuth login redirects to Supabase, but Supabase rejects the callback URL.
**Why it happens:** Supabase validates that the callback URL is in the allowed list.
**How to avoid:** Add `http://localhost:3000/auth/callback` and production URL to Authentication > URL Configuration > Redirect URLs in Supabase dashboard.
**Warning signs:** OAuth redirects fail with "Invalid redirect URL" error.

### Pitfall 2: Browser Client Not Using @supabase/ssr
**What goes wrong:** Client-side auth state doesn't sync with server-side cookies. User appears logged out on page refresh.
**Why it happens:** `createClient` from `@supabase/supabase-js` doesn't manage auth cookies. `createBrowserClient` from `@supabase/ssr` does.
**How to avoid:** Replace `src/lib/supabase/client.ts` to use `createBrowserClient` from `@supabase/ssr`.
**Warning signs:** Auth works on client but server components show unauthenticated.

### Pitfall 3: Missing Middleware Causes Stale Auth Tokens
**What goes wrong:** Server Components get expired auth tokens and fail silently.
**Why it happens:** Server Components can read cookies but cannot write them. Without middleware refreshing tokens, they go stale.
**How to avoid:** Middleware must call `getClaims()` (or `getUser()`) on every request to trigger token refresh and cookie updates.
**Warning signs:** Intermittent auth failures, especially after idle periods.

### Pitfall 4: Profile Trigger Fails Silently on Signup
**What goes wrong:** User signs up but no profile row is created. App crashes when querying profiles.
**Why it happens:** Trigger function errors (column mismatch, permission issues) block the signup or fail silently.
**How to avoid:** Use `SECURITY DEFINER` with empty `search_path`. Test trigger with each signup method (Google, Discord, email). Add `COALESCE` fallbacks for optional metadata fields.
**Warning signs:** Users can log in but profile queries return null.

### Pitfall 5: PostHog Automatic Page Views Fire Incorrectly in App Router
**What goes wrong:** Duplicate page views, missed client-side navigations, or page views before hydration.
**Why it happens:** PostHog's automatic `capture_pageview` was designed for traditional SPAs, not React Server Components.
**How to avoid:** Set `capture_pageview: false` in PostHog init and use a custom `PostHogPageView` component that listens to `pathname` and `searchParams` changes.
**Warning signs:** Page view counts don't match actual navigation, or page views have wrong URLs.

### Pitfall 6: RLS Policies Block Unauthenticated Free Quiz
**What goes wrong:** The one free quiz for unauthenticated users fails because RLS requires `auth.uid()`.
**Why it happens:** If RLS only allows `authenticated` role, `anon` users cannot insert quiz data.
**How to avoid:** Add separate `anon` INSERT policies with `WITH CHECK (user_id IS NULL)`. Allow anon inserts but scope them to null user_id.
**Warning signs:** Free quiz completes but results fail to save.

### Pitfall 7: Middleware Matcher Too Broad or Too Narrow
**What goes wrong:** Either static assets get intercepted (slow), or auth routes are unprotected.
**Why it happens:** Default matcher patterns vary across docs and examples.
**How to avoid:** Use the standard Supabase matcher that excludes `_next/static`, `_next/image`, `favicon.ico`, and common image extensions. Test that `/login` is accessible without auth.
**Warning signs:** Static assets load slowly, or public pages redirect to login.

### PostHog User Identification on Auth State Change

```typescript
// src/lib/posthog/identify.tsx
'use client'

import { useEffect } from 'react'
import { usePostHog } from 'posthog-js/react'
import { createClient } from '@/lib/supabase/client'

export function PostHogIdentify() {
  const posthog = usePostHog()

  useEffect(() => {
    const supabase = createClient()

    const { data: { subscription } } = supabase.auth.onAuthStateChange(
      (event, session) => {
        if (event === 'SIGNED_IN' && session?.user) {
          posthog.identify(session.user.id, {
            email: session.user.email,
          })
        } else if (event === 'SIGNED_OUT') {
          posthog.reset()
        }
      }
    )

    // Check initial state
    supabase.auth.getUser().then(({ data: { user } }) => {
      if (user) {
        posthog.identify(user.id, { email: user.email })
      }
    })

    return () => subscription.unsubscribe()
  }, [posthog])

  return null
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `@supabase/auth-helpers-nextjs` | `@supabase/ssr` | 2024 | Deprecated. No bug fixes. Use SSR package only |
| `getSession()` in server code | `getClaims()` / `getUser()` | 2025 | `getSession()` reads unverified JWT from cookies. Security risk. |
| `getUser()` in middleware | `getClaims()` in middleware | 2025 | `getClaims()` validates JWT locally. Much faster. Use `getUser()` only when you need fresh user data. |
| PostHog `capture_pageview: true` | `capture_pageview: false` + custom component | 2024 | App Router client navigations not captured by default auto-tracking |
| PostHog provider-only init | `instrumentation-client.ts` available | Next.js 15.3+ | Lighter approach, but provider pattern still works and is more flexible for conditional init |

## Open Questions

1. **`getClaims()` behavior with symmetric keys**
   - What we know: `getClaims()` validates JWT locally using JWKS. Supabase projects may use symmetric or asymmetric keys.
   - What's unclear: Whether the installed version handles symmetric keys identically (some docs suggest it falls back to a network call).
   - Recommendation: Test `getClaims()` locally after middleware setup. If it fails, fall back to `getUser()` in middleware.

2. **Free quiz for `/simulated-test`**
   - What we know: CONTEXT.md says one free practice quiz. Simulated test is listed as requiring auth.
   - What's unclear: Whether the free quiz applies to practice mode only or also simulated test.
   - Recommendation: Based on CONTEXT.md wording ("one free practice quiz"), limit free quiz to `/practice` only. Simulated test always requires auth.

3. **PostHog `defaults` option**
   - What we know: PostHog docs mention a `defaults: '2026-01-30'` option that auto-configures recommended settings.
   - What's unclear: Exactly which defaults this sets (could override `capture_pageview` setting).
   - Recommendation: Omit `defaults` and configure explicitly to maintain full control.
